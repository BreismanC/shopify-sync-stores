# Specification: NextAuth + NestJS Hybrid Authentication Flow

## 1. Context

This document describes the authentication flow for `shopify-sync-stored`, specifically the integration between NextAuth.js (Frontend) and NestJS (Backend).

**Current State:**
- NestJS handles credentials and social authentication (Google/Facebook) with Passport.js.
- The frontend receives a JWT via URL parameters after social login and stores it in `localStorage`.

**Objective:**
- Implement NextAuth as the session manager for the frontend client.
- Maintain NestJS as the source of truth for authentication and authorization (issuing and validating tokens).
- Introduce Refresh Tokens to handle JWT expiration gracefully.

## 2. Architecture

### 2.1 Token Management
- **Access Token (JWT):** Generated by NestJS. Used for API authorization (`Bearer Token`). Short-lived (1 hour).
- **Refresh Token:** Generated by NestJS. Used to obtain a new Access Token when the current one expires. Long-lived (7 days), stored in DB.

### 2.2 Responsibilities
- **NestJS:**
  - Validates credentials (email/password) and social providers (Google/Facebook).
  - Issues JWT and Refresh Tokens.
  - Provides endpoint to refresh expired tokens (`/auth/refresh`).
  - Validates the JWT on protected API routes (`@UseGuards(JwtAuthGuard)`).
- **NextAuth (Frontend):**
  - Manages client-side session state.
  - Stores the NestJS JWT and Refresh Token in an encrypted cookie.
  - Handles token refresh automatically using the `jwt` callback.
  - Provides `useSession` and `signIn`/`signOut` hooks to the React components.

### 2.3 Flow Diagram

#### A. Social Login (Google/Facebook)
```
[User] -> [Frontend] "Login with Google"
      -> [NestJS: GET /api/auth/google]
      -> [Google OAuth]
      -> [NestJS: /api/auth/google/callback]
         - AuthService.login(user) generates:
           - access_token (JWT, 1h)
           - refresh_token (7d, saved in DB)
      -> [Redirect: /auth/callback?token=<jwt>&refresh_token=<rt>&user=<json>]
      -> [Frontend: AuthCallbackPage]
         - Calls signIn('credentials', { token, refreshToken, user })
         - NextAuth stores tokens in encrypted session cookie
      -> [Redirect to /dashboard or /onboarding based on tenantId]
```

#### B. Credentials Login
```
[User] -> [Frontend] Login Form
      -> [NestJS: POST /api/auth/login]
         - Validates credentials
         - Generates access_token + refresh_token
      -> [Frontend receives { access_token, user }]
      -> [Calls signIn('credentials', { token, refreshToken, user })]
```

#### C. API Calls (Protected Routes)
```
[Component] -> [SWR Hook] -> [fetchWithAuth wrapper]
      -> [NestJS: /api/protected-route]
         - Header: Authorization: Bearer <jwt>
         - JwtAuthGuard validates token
      -> [Response 200/401]
```

#### D. Token Refresh Flow
```
[JWT Expiration Check]
      - NextAuth `jwt` callback decodes token.exp
      - If exp < now + 5min:
         - Fetch /api/auth/refresh with refresh_token
         - If success: Update session with new tokens
         - If fail: Set session.error = "RefreshAccessTokenError"
```

## 3. NextAuth Configuration

**File:** `apps/frontend/app/api/auth/[...nextauth]/route.ts`

### 3.1 Providers
- **Credentials Provider:** Used for all logins (social data passed as credentials after OAuth callback).
  - Credentials contain: `token`, `refreshToken`, `user` (JSON string).

### 3.2 Callbacks
- **jwt:** Intercepts token creation/update.
  - On Login: Stores `accessToken`, `refreshToken`, and `user` data in the NextAuth JWT.
  - On Update: Checks `accessToken.exp` every request. If expired/near expiration, calls backend refresh endpoint.
- **session:** Passes `accessToken` and user data to the client-side session object accessible via `useSession()`.

### 3.3 Pages
- Custom Login: `/auth/login`
- Error: `/auth/error`

## 4. Backend Changes (NestJS)

### 4.1 Refresh Token Implementation
- **Entity:** Add `RefreshToken` table or column in `User` entity (store hash of the token).
- **Service:** `AuthService.generateTokens(user)` must return `access_token` and `refresh_token`.
- **Endpoint:** `POST /api/auth/refresh`
  - Input: `{ refreshToken: string }`
  - Process: Find user by token hash, validate expiration.
  - Output: `{ access_token: string, refresh_token: string }` (optionally rotate refresh token).

### 4.2 User Entity Update
- Add fields for refresh token storage (e.g., `refreshTokenHash`, `refreshTokenExpiry`).

## 5. Frontend Changes (Next.js)

### 5.1 NextAuth Setup
- Install `next-auth`.
- Create `[...nextauth]/route.ts` with `CredentialsProvider`.

### 5.2 Auth Callback Page Update
- **File:** `apps/frontend/app/auth/callback/page.tsx`
- Instead of `localStorage.setItem('token', ...)`, call `signIn('credentials', ...)`.
- Extract query params (`token`, `refresh_token`, `user`).

### 5.3 SWR + Auth Integration
- Create `fetchWithAuth` (wrapper using SWR or standard fetch with interceptor).
- **Note:** `fetchWithAuth` must check `useSession()` to get `data?.accessToken`.

## 6. Data Model

### 6.1 User Entity (NestJS)
```typescript
@Entity()
export class User {
  // ... existing fields ...
  @Column({ nullable: true })
  refreshTokenHash: string; // Store hash of the refresh token
  @Column({ type: 'timestamp', nullable: true })
  refreshTokenExpiresAt: Date;
}
```

## 7. Security Considerations
- **Refresh Token Rotation:** Ideally, issue a new refresh token with every refresh (rotation) to prevent replay attacks.
- **HttpOnly Cookies:** NextAuth uses HttpOnly cookies by default for the session, which is safer than localStorage.
- **CORS:** Ensure NestJS CORS is configured to allow requests from the Next.js domain.
